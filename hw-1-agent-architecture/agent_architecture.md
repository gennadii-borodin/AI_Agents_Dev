### 1. Высокоуровневая архитектура (Схема)

Система построена по паттерну **Orchestrator + Specialized Tools/Agents**. Главный агент (Оркестратор) управляет циклом ReAct и вызывает специализированные инструменты, которые инкапсулируют в себе вызовы разных LLM.

```text
[ Пользователь / CI/CD Pipeline ]
           │ (Триггер: ID тест-кейса в TestIT)
           ▼
┌─────────────────────────────────────────────────────────┐
│                 ORCHESTRATOR AGENT (ReAct)              │
│  [Модель: Claude 3.5 Sonnet / GPT-4o] (Планирование)    │
│  - Декомпозиция задачи                                  │
│  - Выбор инструментов (Tool Calling)                    │
│  - Агрегация результатов                                │
└────────┬──────────┬──────────┬──────────┬───────────────┘
         │          │          │          │
   [Tool 1]     [Tool 2]   [Tool 3]   [Tool 4]
   Fetch Data   Check      Evaluate   Generate
   (TestIT &    Format     Test       Report
   Confluence)  & Rules    Design     (JSON/MD)
         │          │          │          │
         ▼          ▼          ▼          ▼
   ┌──────────┐ ┌────────┐ ┌────────┐ ┌────────┐
   │ API      │ │ LLM    │ │ LLM    │ │ LLM    │
   │ Wrapper  │ │ Tier 1 │ │ Tier 3 │ │ Tier 2 │
   │ (Python) │ │(Qwen7B)│ │(Sonnet)│ │(4o-mini│
   └──────────┘ └────────┘ └────────┘ └────────┘
         │          │          │          │
         ▼          ▼          ▼          ▼
   [TestIT API] [Regex/AST]  [Vector DB] [TestIT API]
   [Confluence] [JSON Schema][RAG Rules]  [Write Comment]
```

### 2. Цикл работы агента (Reason → Act → Observe)

Агент работает в цикле ReAct. Ниже приведен пример того, как Оркестратор обрабатывает один тест-кейс.

#### Шаг 1: Сбор контекста
*   **Reason (Мысль):** "Мне поступил ID тест-кейса `TEST-102`. Чтобы провести ревью, мне нужно получить его тело из TestIT и найти связанные требования в Confluence."
*   **Act (Действие):** Вызов инструмента `get_test_context(test_id="TEST-102")` и `search_requirements_in_confluence(test_id="TEST-102")`.
*   **Observe (Наблюдение):** Инструменты вернули JSON с шагами теста, ожидаемыми результатами и текст требования из Confluence (ID `REQ-404`).

#### Шаг 2: Проверка регламентов (Роутинг на дешевую модель)
*   **Reason:** "Теперь нужно проверить, соответствует ли тест корпоративному шаблону (наличие предусловий, шагов, постусловий, ожидаемых результатов)."
*   **Act:** Вызов инструмента `check_template_compliance(test_json)`. *Внутри инструмента вызывается дешевая модель (Tier 1).*
*   **Observe:** Инструмент вернул: "Отсутствует блок 'Постусловия'".

#### Шаг 3: Анализ тест-дизайна (Роутинг на продвинутую модель)
*   **Reason:** "Требование `REQ-404` описывает поле 'Возраст' от 18 до 99. Тест-кейс проверяет только значение 25. Мне нужно оценить полноту покрытия и применение техник тест-дизайна (граничные значения, классы эквивалентности)."
*   **Act:** Вызов инструмента `evaluate_test_design(requirement_text, test_steps)`. *Внутри инструмента вызывается продвинутая модель (Tier 3) с промптом, содержащим правила тест-дизайна.*
*   **Observe:** Модель вернула критику: "Не покрыты граничные значения (17, 18, 99, 100) и негативные сценарии (буквы, спецсимволы)".

#### Шаг 4: Формирование отчета
*   **Reason:** "У меня есть все данные для формирования финального ревью. Нужно сгенерировать структурированный фидбек для автора теста."
*   **Act:** Вызов `generate_review_report(findings)`.
*   **Observe:** Сформирован Markdown-отчет.
*   **Act:** Запись отчета в комментарий к тест-кейсу в TestIT через API.

### 3. Логика роутинга и Выбор моделей

Чтобы оптимизировать бюджет и latency, применяется **Multi-Tier Model Routing**.

| Уровень (Tier) | Задача | Модель | Обоснование выбора |
| :--- | :--- | :--- | :--- |
| **Tier 0** | Структурная валидация, поиск по регулярным выражениям, проверка JSON-схемы. | **Python Code / Regex** | Не требует LLM. Работает мгновенно, стоит $0. |
| **Tier 1** | Проверка формата, извлечение сущностей, суммаризация коротких текстов, проверка наличия обязательных полей. | **Qwen 2.5 7B / Llama 3 8B** (или **GPT-4o-mini**) | Дешевые, быстрые модели. Отлично справляются с инструкциями "найди/проверь формат". Хватает контекстного окна и логики. |
| **Tier 2** | Оркестрация, планирование, сложный маппинг требований к тестам, генерация финального отчета. | **GPT-4o / Claude 3.5 Haiku** | Хороший баланс цены и качества для задач рассуждения и структурирования. |
| **Tier 3** | Глубокий анализ тест-дизайна, поиск логических дыр, генерация недостающих тест-кейсов, сложная дедукция. | **Claude 3.5 Sonnet / GPT-4o** | Лучшие модели для сложного логического рассуждения (CoT), понимания нюансов бизнес-логики и техник тест-дизайна. |

### 4. Декомпозиция задачи (Подзадачи агента)

1.  **Context Retrieval (Сбор данных):** Парсинг TestIT, извлечение ID требований, поиск страниц в Confluence.
2.  **Structural Validation (Структура):** Проверка наличия всех обязательных секций (Предусловия, Шаги, Оживаемый результат, Постусловия).
3.  **Traceability Check (Трассируемость):** Проверка, что каждый шаг теста ссылается на конкретное требование (нет "висячих" тестов).
4.  **Test Design Evaluation (Тест-дизайн):** Анализ применения Equivalence Partitioning, Boundary Value Analysis, State Transition.
5.  **Coverage Analysis (Покрытие):** Проверка, что все acceptance criteria из требования покрыты тестами (требует анализа не одного теста, а чек-листа/набора тестов).
6.  **Feedback Generation (Фидбек):** Формирование вежливого, конструктивного и структурированного комментария.

### 5. Механизмы защиты (Guardrails)

#### Защита от Dead Loop (Зацикливания)
Агент может застрять, пытаясь найти несуществующее требование в Confluence или переписывая один и тот же шаг.
*   **Circuit Breaker (Максимум итераций):** Жесткий лимит в 5-7 шагов ReAct. Если лимит достигнут, агент прерывает работу и возвращает статус `NEEDS_HUMAN_REVIEW` с дампом текущего состояния.
*   **State Tracking:** Оркестратор ведет список `visited_tools` и `visited_entities`. Если агент пытается вызвать `search_confluence` с тем же запросом второй раз, система блокирует вызов и заставляет сменить стратегию.
*   **Fallback:** При превышении лимита токенов на генерацию или ошибок API, система эскалирует задачу сеньору.

#### Защита от исчерпания токенов (Context Window Overflow)
Страницы в Confluence могут быть огромными (100k+ токенов).
*   **Map-Reduce для Coverage:** Если нужно проверить покрытие эпика (много требований), агент сначала суммаризирует каждое требование (Tier 1), а затем анализирует матрицу покрытия (Tier 2).
*   **Strict Output Schema:** Использование JSON mode / Structured Outputs (OpenAI) или Tool Calling, чтобы модель не лила "воду", а выдавала только факты.

### 6. Системный промпт (Оркестратор)

```text
Ты — Senior QA Automation и Архитектор тестирования. Твоя задача — провести автоматическое кросс-ревью тест-кейса.
Ты работаешь в цикле Reason -> Act -> Observe.

ПРАВИЛА И РЕГЛАМЕНТЫ КОМПАНИИ:
1. Каждый тест-кейс ОБЯЗАН содержать секции: [Предусловия], [Шаги], [Оживаемый результат], [Постусловия].
2. Граничные значения должны проверяться явно (например, для диапазона 1-10 нужны тесты на 0, 1, 10, 11).
3. Тест не должен дублировать логику другого теста.
4. В шагах нельзя использовать размытые формулировки ("ввести данные"), должны быть конкретные значения.

ДОСТУПНЫЕ ИНСТРУМЕНТЫ (TOOLS):
- `get_test_data(test_id)`: Возвращает JSON с телом теста из TestIT.
- `get_requirement_context(req_id)`: Возвращает релевантный кусок текста из Confluence.
- `check_formatting(test_json)`: [Tier 1] Проверяет наличие обязательных полей.
- `evaluate_test_design(req_text, test_steps)`: [Tier 3] Глубоко анализирует техники тест-дизайна.
- `post_review_comment(test_id, markdown_report)`: Сохраняет ревью в TestIT.

АЛГОРИТМ РАБОТЫ:
1. Получи данные теста.
2. Определи связанные требования и получи их контекст.
3. Вызови check_formatting.
4. Вызови evaluate_test_design.
5. Сформируй итоговый отчет в формате Markdown, где укажи:
   - [PASS] Что сделано хорошо.
   - [FAIL] Критические ошибки (ссылка на пункт регламента).
   - [WARNING] Рекомендации по улучшению.
6. Опубликуй отчет через post_review_comment.

ОГРАНИЧЕНИЯ:
- Не выдумывай требования. Если их нет в Confluence, пиши "Требование не найдено".
- Максимум 10 вызовов инструментов. Если не можешь найти данные, остановись и запроси помощь человека.
- Отвечай строго на русском языке.
```

### 7. Приблизительная экономика (Расчет стоимости)

Предположим, команда создает **200 тест-кейсов в месяц**.
Средний размер теста: ~400 токенов. Средний размер контекста требования (после RAG): ~1500 токенов.
Всего входных токенов на 1 ревью: ~2000. Выходных (отчет + рассуждения): ~1000.

*Расчет на 1 ревью (1 тест):*

| Этап | Модель | Input Tokens | Output Tokens | Стоимость (Input) | Стоимость (Output) | Итого за 1 тест |
| :--- | :--- | :--- | :--- | :--- | :--- | :--- |
| Оркестратор (ReAct) | GPT-4o | 2,000 | 500 | $0.010 | $0.0075 | $0.0175 |
| Проверка формата | GPT-4o-mini | 500 | 100 | $0.000075 | $0.00006 | $0.000135 |
| Анализ тест-дизайна | Claude 3.5 Sonnet | 2,000 | 400 | $0.006 | $0.006 | $0.012 |
| **ИТОГО на 1 тест** | | | | | | **~$0.03** |

**Экономика на месяц (200 тестов):**
*   **Стоимость API:** 200 * $0.03 = **$6 в месяц**.
*   **Время Senior QA:** Ручное ревью одного теста занимает ~15 минут. 200 тестов = 50 часов. При ставке сеньора $30/час, экономия составляет **$1,500 в месяц**.
*   **ROI:** ~5000%.
